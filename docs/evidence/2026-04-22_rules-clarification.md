## Ruling 1 — "Start from Scratch" Requirement

**Thread title**: Strict from scratch requirement conflict
**OP**: anton (participant)
**OP timestamp**: 2026-04-22 06:06 KST

> How strict it is to start completely from scratch with this hackathon
> work, because it's kind of blurry. When we submitted the hackathon
> idea, chatbot replied that it's not mandatory to start completely
> from scratch, meaning that, for example, some ground-up, like a
> repository with something could already exist. We had been asked to
> be experts in our area and build the hackathon work on top of what
> we are working with and what we are experts in. So I assume we could
> proceed with the topic that was submitted when we applied for the
> hackathon, right? Hopefully, the assessment will be for the work
> done during hackathon time, which must be huge, but could be on top
> of some ground work done a bit in advance.

**Moderator response** — Ado | Claude (Anthropic official moderator)
**Timestamp**: 2026-04-22 06:18 KST

> Hey @anton — the requirement is to build a new project with valid
> GitHub commits from when the hackathon started to when it ends.
>
> That's not to say you can't take inspiration and learnings from
> prior works into building the hackathon project, but the code for
> the hackathon needs to be written this week.
>
> This is to keep it fair for all participants.

### Operative rule (Rhythmscape interpretation)

1. **Code cut-off**: All implementation code must have valid GitHub
   commits *after* hackathon kickoff (2026-04-22 01:00 KST).
2. **Prior works permitted**: "Inspiration and learnings from prior
   works" are allowed — this covers theoretical frameworks,
   specification documents, literature scouting, and prompt design
   rooted in the researcher's area of expertise.
3. **Fairness constraint**: The rule exists to equalize participants,
   not to penalize domain expertise. Declaring prior planning work
   transparently (see Transparency Note in README) is the
   rule-compliant path.

## Ruling 2 — Personal API Credits Beyond the $500 Allowance

**Thread title**: Personal API credits
**OP**: Lucas76hz (participant)
**OP timestamp**: 2026-04-21 23:36 KST

> Hi moderators! Quick question regarding the API credits:
>
> There has been some discussion among participants about whether teams
> are allowed to use personal funds to purchase additional API credits
> once the hackathon-provided ones run out. Since this could
> significantly impact the scale of the projects and the overall
> fairness of the competition, could we get an official ruling on this?
> Are teams allowed to use personal accounts/funds for extra compute,
> or are we strictly limited to the provided credits?

**Response** — Joshua Jerin | Cerebral Valley (co-host)
**Timestamp**: 2026-04-21 23:52 KST

> You can use how many ever credits you want! You will however receive
> only $500 of API credits as part of the hackathon tho

### Operative rule (Rhythmscape interpretation)

1. **Personal funds permitted**: Participants may purchase additional
   API credits beyond the hackathon-provided $500 allowance.
2. **Responder note**: Joshua Jerin is a Cerebral Valley (co-host)
   moderator, not an Anthropic staff member. The ruling received
   community reactions (👍 3, ❤️ 1) and has not been contradicted.
   Treat as authoritative but secondary to Anthropic staff rulings
   (e.g., Ado | Claude) if a conflict arises.
3. **Strategic implication**: The 4-city scale-up (Changwon flagship
   + Busan-Yeongdo + Jinju + Sacheon) with Managed Agents batch
   processing can proceed without artificial credit-budget throttling.
